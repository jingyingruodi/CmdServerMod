# 多人游戏架构设计：根本性问题分析

## 问题描述

你的观察指出了一个关键的架构问题：

> "日志能反映出消息底层是广播的但是mod有效给拦截了，日志能看到其他玩家的输入"

这表明**当前的设计本质上是错误的**。

---

## 架构对比分析

### ❌ 当前实现（错误的方式）

```
玩家A (客户端)                           玩家B (客户端)
    │                                       │
    ├─ 输入 /help                          │
    │                                       │
    └─→ 通过网络广播给所有客户端           │
        │                                   │
        ├─→ 玩家A的客户端接收 [CHAT]       ├─→ 玩家B的客户端接收 [CHAT]
        │   ├─ [Simulation] 识别为命令        ├─ [Simulation] 识别为命令
        │   ├─ [Simulation] 执行命令          ├─ [Simulation] 执行命令
        │   ├─ [Simulation] 标记结果          ├─ [Simulation] 标记结果
        │   └─ [UI] 过滤：仅显示给A          └─ [UI] 过滤：不显示（不是给B的）
        │
        └─→ 问题：每个客户端都在执行服务器逻辑！
            - 重复计算
            - 可以被客户端修改
            - 不公平（有权限系统时，客户端可以绕过）
            - 不安全
```

### ✅ 正确的方式（中心化管理）

```
玩家A (客户端)          服务端/房主          玩家B (客户端)
    │                     │                     │
    ├─ 输入 /help         │                     │
    │   │                 │                     │
    │   └─→ 发送给服务端  │                     │
    │                 ┌───┴───┐                 │
    │                 │       │                 │
    │            [Simulation] 识别为命令         │
    │            [Simulation] 执行命令           │
    │            [Simulation] 生成结果           │
    │                 │       │                 │
    │                 └───┬───┘                 │
    │                     │                     │
    │        ┌────────────┴──────────┐          │
    │        │                       │          │
    │    返回给玩家A              返回给玩家B   │
    │    命令结果                  无内容
    │        │                       │
    └────────┴───────────────────────┘
    
优势：
    ✅ 命令在中心（服务端）执行，不可篡改
    ✅ 权限检查在服务端进行，不可绕过
    ✅ 只有有权限的玩家才能看到信息
    ✅ 符合游戏设计规范
```

## 问题的三个方面

### 1. 架构设计问题（最严重）

**当前**：Simulation 层既处理命令执行，也处理过滤
```lua
-- 这是错误的分层
[Simulation] 执行命令 → [UI] 过滤结果
```

**应该是**：Simulation 层只在服务端执行，结果由服务端决定发送给谁
```lua
-- 这是正确的分层
[服务端 Simulation] 执行命令 → [服务端决定] 发送给谁 → [客户端接收并显示]
```

### 2. 安全性问题（严重）

**漏洞**：客户端可以修改 UI 层逻辑
```lua
-- 玩家可以修改 ui/main.lua，移除过滤逻辑
-- 然后看到所有其他玩家的命令结果
```

**正确做法**：
```lua
-- 服务端决定发送什么
-- 客户端只是被动接收和显示
-- 没有办法修改（无法修改服务端代码）
```

### 3. 游戏逻辑问题（隐蔽但危险）

**当前**：所有客户端都在执行 `/settings` 等命令
```lua
-- 每个客户端都调用 Map.GetSettings()
-- 多人时，玩家数×命令次数的重复计算
```

**正确方式**：
```lua
-- 只有服务端/房主执行
-- 其他客户端接收结果并显示
```

## 应该如何做？

### 第一步：理解 Desynced 的多人架构

Desynced 是个多人游戏，有几种可能的运行模式：

1. **单机游戏**（房主模式）
   - 主机是房主，同时是玩家
   - 房主的计算机是"服务端"
   - 其他加入的玩家是"客户端"

2. **专用服务端**（我们的研究对象）
   - 独立的服务器运行游戏逻辑
   - 所有玩家都是"客户端"

3. **点对点**（可能存在）
   - 每个客户端都有完整的游戏状态副本

### 第二步：确定当前模式

根据 `def.json` 中我们定义的两个 Package：
```json
"packages": {
    "UI": { ... },        // 仅在有 UI 的环境运行
    "Simulation": { ... } // 在所有环境运行
}
```

这说明：
- **房主/单机**：两个 Package 都运行
- **客户端加入房主游戏**：也是两个都运行（包括 Simulation）
- **专用服务端**：...

**这里有个关键问题**：我们没有定义 Data Package！

```json
// 应该有
"packages": {
    "Data": { ... }           // 游戏逻辑和地图数据
    "UI": { ... }             // UI（仅客户端）
    "Simulation": { ... }     // 命令系统（所有环境）
}
```

### 第三步：重新设计架构

#### 对于房主模式（玩家直接加入）

```
房主的PC：
  [Data] - 游戏逻辑
  [Simulation] - 命令处理（也运行）
  [UI] - 显示界面

客户端加入时会同步：
  [Data] - 游戏逻辑（同步状态）
  [Simulation] - 命令处理（也运行！）
  [UI] - 显示界面

问题：两边都在跑 Simulation，容易不同步
```

#### 对于专用服务端

```
服务端：
  [Data] - 游戏逻辑
  [Simulation] - 命令处理
  [UI] - 无（server 模式无 UI）

客户端：
  [Data] - 游戏逻辑（同步状态）
  [Simulation] - 不应该有！（或者应该被禁用）
  [UI] - 显示界面
```

## 正确的实现方式

### 原则

1. **服务端决定一切**
   - 命令执行在服务端进行
   - 权限检查在服务端进行
   - 结果由服务端决定如何分发

2. **客户端被动接收**
   - 客户端不执行命令逻辑
   - 客户端只接收服务端的结果
   - 客户端负责显示结果

3. **消息流向清晰**
   ```
   客户端 → 网络 → 服务端[执行] → 网络 → 客户端[显示]
   ```

### 实现策略

#### 方案 A：使用游戏的内置消息系统

```lua
-- simulation/main.lua (在服务端运行)

-- 如果是房主或专用服务端
if Game.IsServer() then  -- 检查是否为服务端/房主
    -- 在服务端执行命令
    function Chat.Text(arg, player_id)
        if is_command(arg.txt) then
            local result = execute_command(arg.txt, player_id)
            -- 通过游戏的网络系统发送给特定玩家
            -- （需要查游戏 API 如何实现）
        end
    end
else
    -- 在客户端，不执行命令，只显示
    -- Simulation 层不应该有命令处理
end
```

#### 方案 B：使用 UI 层的消息系统

```lua
-- 服务端在 Data Package 中定义自定义消息
-- 客户端订阅并显示这些消息
```

## 当前实现的具体问题

### 问题 1：Simulation 在所有环境都运行

```lua
-- 当前：房主的客户端和加入的客户端都运行相同的 Simulation
-- 结果：所有玩家都在执行命令，容易不同步
```

### 问题 2：没有真正的服务端逻辑

```lua
-- 当前：我们在客户端 hook UI.Run()，修改消息
-- 应该是：服务端执行命令，客户端接收结果
```

### 问题 3：权限系统无法实现

```lua
-- 当前的客户端可以修改 ui/main.lua 来移除过滤
-- 无法实现真正的权限检查
```

## 建议的解决方向

### 短期（v1.0.2）

**状态**：确认当前设计的局限性

1. 在文档中明确说明：
   ```
   当前模组仅适用于 [房主/多人游戏]
   不适用于 [专用服务端]（或需要特殊配置）
   ```

2. 添加 `Game.IsServer()` 或类似的检测

3. 在房主模式下运行正常的命令系统

### 中期（v1.1）

**重构**：引入正确的分层

1. 查研 Desynced 的多人 API
   - 是否有 `Game.IsServer()` 函数？
   - 如何在服务端和客户端之间发送自定义消息？
   - 专用服务端的限制是什么？

2. 根据 API 设计正确的架构

3. 完全重写消息处理逻辑

### 长期（v1.2+）

**完整的权限系统**：
- 仅在服务端执行
- 客户端无法绕过
- 支持细粒度权限控制

## 问题的根本原因

这个问题之所以存在，是因为：

1. **模组开发早期**，直接修改了 UI 层，看起来能工作
2. **双端兼容需求**，导致逻辑放在了 Simulation（两端都运行）
3. **没有研究 Desynced 的多人 API**，不清楚怎么才能真正做到"服务端中心化"

---

## 总结

**你的观察是正确的**：

当前的架构违反了现代网络游戏的基本原则。正确的做法应该是：

1. **服务端/房主执行命令**
2. **结果由服务端发送给客户端**
3. **客户端只负责显示**

这需要深入了解 Desynced 的多人通信 API，然后重新设计。

现在的实现虽然在单机或房主模式下"看起来能工作"，但在安全性、公平性和可维护性上都有根本缺陷。
