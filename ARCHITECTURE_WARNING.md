# 🚨 架构问题警告

**最后更新**：2026-03-20

## 当前状态

⚠️ **重要**：项目的架构存在根本性问题，**不符合现代网络游戏的最佳实践**。

---

## 核心问题

### 问题陈述

当前实现中：

```
日志显示：
  玩家A的客户端 → 执行命令
  玩家B的客户端 → 也执行命令（同样的代码）
  玩家C的客户端 → 也执行命令（同样的代码）

这是错误的！应该是：
  玩家A/B/C → 发送给房主/服务端
  房主/服务端 → 执行命令一次
  房主/服务端 → 决定将结果发送给谁
```

### 为什么这是问题？

1. **重复计算**  
   - 相同的命令在每个客户端上执行一次
   - 浪费资源和带宽

2. **不可靠的权限**  
   - 权限检查在客户端进行（客户端可以修改）
   - 无法真正实现权限系统

3. **安全隐患**  
   - 玩家可以修改 `ui/main.lua` 来移除过滤逻辑
   - 看到其他玩家的命令结果
   - 修改命令执行逻辑

4. **不一致的状态**  
   - 在房主模式下，房主和客户端都在执行逻辑
   - 容易导致状态不同步

---

## 当前工作的场景

✅ **房主模式（玩家直接加入房主游戏）**
- 虽然架构不对，但"碰巧"工作了
- 只要房主和客户端运行相同版本的模组
- 即使有问题也能看起来工作正常

❌ **专用服务端**  
- 架构完全错误
- 服务端没有 UI（无法过滤结果）
- 需要完全不同的实现

---

## 正确的做法

### 原则

```
房主/服务端 → [执行命令] → [生成结果] → [决定发送给谁]
      ↓
    网络
      ↓
客户端 → [接收结果] → [显示在 UI]
```

### 关键特性

1. **服务端中心化**
   - 命令只在服务端执行一次
   - 结果由服务端决定分发

2. **客户端被动**
   - 客户端只接收服务端的结果
   - 客户端无法修改执行逻辑

3. **权限系统安全**
   - 权限检查在服务端
   - 客户端无法绕过
